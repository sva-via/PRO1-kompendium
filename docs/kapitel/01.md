# Kapitel 1 – Hvad er et objekt?

## 1. Introduktion

Dette kapitel handler om det mest grundlæggende spørgsmål i objektorienteret programmering: Hvad er et objekt?

I PRO1 er målet ikke at lære Java som sprog, men at lære at modellere virkeligheden i software. Et objekt er ikke blot en teknisk konstruktion – det er en model af et begreb fra et domæne.

Vi begynder derfor ikke med syntaks, men med forståelse:
- Hvad er forskellen på en variabel og et objekt?
- Hvad betyder det, at en klasse modellerer et begreb?
- Hvad er state og adfærd?

Det gennemgående eksempel i kapitlet er klassen Name.

## 2. Objekt vs. variabel

En variabel er et sted i programmet, hvor vi gemmer en værdi.

int age = 20;

Her gemmer variablen age tallet 20.

Men en variabel er ikke et begreb. Den er kun en beholder.

Et objekt derimod repræsenterer noget med:
- En tilstand (state)
- En adfærd (metoder)
- Et ansvar

Name name = new Name("Ada", "Lovelace");

Her er name en variabel – en reference til et Name-objekt
Men objektet er instansen af Name.

Variablen er teknisk. Objektet er begrebsmæssigt.

## 3. Klassen som model af et begreb

En klasse beskriver, hvordan objekter af en bestemt type ser ud og opfører sig.

public class Name {

    private String firstName;
    private String lastName;

    public Name(String firstName, String lastName) {
        this.firstName = firstName;
        this.lastName = lastName;
    }

    public String getFullName() {
        return firstName + " " + lastName;
    }
}

Klassen modellerer begrebet navn. Den indeholder state og adfærd og afgrænser sit ansvar.

## 4. State og adfærd

State er de data, objektet indeholder.
Adfærd er det, objektet kan gøre.

State i Name:
private String firstName;
private String lastName;

Adfærd:
public String getFullName() {
    return firstName + " " + lastName;
}

Adfærd bør være relateret til objektets begreb.

## 5. Gennemgående eksempel: Name

Eksempel på yderligere metoder:

public String getFirstName() {
    return firstName;
}

public String getLastName() {
    return lastName;
}

public boolean hasSameLastName(Name other) {
    return this.lastName.equals(other.lastName);
}

Objekter samarbejder via deres offentlige (public) metoder.

## 6. Designovervejelser

Hvorfor lave en klasse for Name?
- Samler relaterede data
- Giver begrebet identitet
- Kontrollerer korrekt brug

Alternativet med løse String-variabler giver ingen samlet struktur eller ansvar.

## 7. Typiske fejl og misforståelser

1) At tro at objekt = variabel
2) At placere adfærd uden for objektet
3) At lade klassen gøre for meget

Et objekt bør repræsentere ét begreb med klart ansvar.

## 8. Refleksionsspørgsmål

1) Hvorfor er det vigtigt at skelne mellem variabel og objekt?
2) Hvad betyder det, at en klasse modellerer et begreb?
3) Hvilken adfærd hører naturligt til i Name?

## 9. Små øvelser

Øvelse 1:
Implementér getInitials() i Name.
Den skal returnere fx "A.L." for "Ada Lovelace".

Overvej:

- Hvem har ansvaret for at danne initialer?

- Skal metoden ligge i Name eller et andet sted?

Øvelse 2:
Vurder designet:
public class Person {
    public String firstName;
    public String lastName;
}

Hvad mister vi ved public felter?

## Afslutning

Et objekt er ikke blot en datastruktur.

Det er:

- En model af et begreb

- En samling af state

- En enhed med ansvar

- En aktør med adfærd

At programmere objektorienteret betyder derfor ikke primært at skrive metoder.

Det betyder at træffe designbeslutninger om:

- Hvilke begreber findes i domænet?

- Hvad er deres ansvar?

- Hvordan skal de samarbejde?

Resten af kurset bygger på denne forståelse.
