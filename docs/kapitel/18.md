# Kapitel 18 – AI som sparringspartner

## 1. Introduktion

I dette kapitel ser vi på, hvordan AI kan bruges som faglig sparringspartner i PRO1. AI er ikke en erstatning for designforståelse, men et værktøj til at udfordre, forbedre og analysere egne løsninger.

Fokus er på tre aktiviteter:
- Generere alternative løsninger
- Identificere fejl i AI-genereret kode
- Forklare og vurdere AI’s designvalg

## 2. Generér alternativ løsning

AI kan bruges til at generere en alternativ implementering af en klasse, fx en anden version af MyDate.

```
public void stepForwardOneDay() {

    int max = daysInMonth(year, month);

    if (day < max) {
        day++;
        return;
    }

    day = 1;

    if (month < 12) {
        month++;
    } else {
        month = 1;
        year++;
    }
}
```
Som studerende bør du spørge: Er denne version mere læsbar end den oprindelige? Bevarer den invarianten i MyDate? Er ansvar tydeligt placeret?

AI kan producere korrekt kode – men vurderingen er din opgave.

## 3. Find fejl i AI-kode

AI-genereret kode kan indeholde subtile fejl. Overvej følgende Clock-implementering:

```
public void tick() {
    second++;
    if (second == 60) {
        minute++;
    }
}
```
Her mangler normalisering af second og hour. Invarianten for Clock brydes, fordi second kan blive 60.

Spørg: Hvilken invariant gælder for Clock? Håndhæves den konsekvent i alle metoder?

At kunne finde fejl i AI-kode er en central kompetence.

## 4. Forklar AI-valg

AI vælger ofte bestemte strukturer (fx if/else frem for switch). Det er vigtigt at kunne forklare disse valg – også hvis du selv ikke ville have valgt dem.

```
private boolean isLeapYear(int year) {
    return (year % 4 == 0 && year % 100 != 0) 
           || (year % 400 == 0);
}
```
Denne kompakte version er korrekt, men mindre eksplicit end en sekventiel if/else-struktur.

Spørg: Hvilken version er mest læsbar for målgruppen i PRO1? Hvilken understøtter tydeligst designbeslutninger?

## 5. Designovervejelser

AI bør bruges til:
- At generere alternative perspektiver
- At få forslag til refactoring
- At få forklaringer på eksisterende kode

AI bør ikke bruges til:
- At erstatte analyse
- At omgå forståelse
- At levere løsninger uden vurdering

Brug AI dialogisk: Stil præcise spørgsmål og kræv forklaring af valg.

## 6. Typiske fejl og misforståelser

1) At acceptere AI-kode ukritisk.
2) At bruge AI før man selv har forsøgt at modellere problemet.
3) At fokusere på syntaks frem for design.
4) At undlade at teste AI-genereret kode.

## 7. Refleksionsspørgsmål

1) Hvordan kan AI hjælpe med at forbedre cohesion og reducere coupling?
2) Hvilke typer fejl ser du typisk i AI-genereret kode?
3) Hvordan kan du dokumentere, at du har forstået AI’s forslag?

## 8. Små øvelser

Øvelse 1: Bed AI generere en alternativ version af Clock med totalSeconds. Sammenlign med din egen løsning og identificér forskelle.

Øvelse 2: Giv AI en bevidst forkert implementering og bed den finde fejlen. Vurdér om forklaringen er korrekt.
